## Problem Statement

30-40대 맞벌이 부부가 가구 전체 자산을 한눈에 파악하기 어렵다. 부동산(소유/전세), 주식(5개 이상 증권사), 연금(IRP/국민연금/퇴직연금), ISA, 우리사주, 금 현물, 코인 등 자산이 10개 이상 계좌에 분산되어 있고, 전체 그림을 보려면 6개 이상 앱에 각각 로그인해야 한다. 한 번 확인에 15-20분 소요, 결과적으로 자주 확인하지 않게 되어 재무 의사결정이 감으로 이루어진다.

## Demand Evidence

- 창업자 본인이 직접 겪는 문제. 부동산 2건, 증권 계좌 10개 이상, 금, 코인까지 관리 중.
- 현재 해결책: 각 앱에 하나씩 로그인하여 확인 (15-20분/회).
- 뱅크샐러드/핀크 등 기존 마이데이터 앱이 있지만, 부동산 전세/매매 구분, 금 현물, 우리사주, 코인 등은 제대로 커버하지 못함.
- 세컨드 오피니언: "15-20분 걸리는데도 하고 있다면 수요는 진짜."

## Status Quo

현재 방법: 키움, 현대차증권, 신한, 하나, 한국투자, 업비트 등 각 앱에 개별 로그인하여 확인. 부동산 시세는 네이버 부동산/호갱노노 별도 확인. 부부 간 자산 공유 수단 없음.

비용: 시간(15-20분/회), 빈도 저하(자주 안 하게 됨), 부부 간 정보 비대칭.

## Target User & Narrowest Wedge

**타겟:** 30-40대 맞벌이 부부. 자산이 여러 금융기관에 분산, 부부 각자 계좌 관리, 전체 가구 자산 파악 어려움.

**Narrowest Wedge:** "종목/수량만 입력하면 시세가 자동 반영되는 가구 자산 대시보드." 한번 설정하면 시세는 공개 API로 자동 업데이트. 부부가 한 대시보드를 공유.

## Constraints

1. 마이데이터 사업자 등록 없이 구현 (금융위 허가 불필요)
2. 증권사 직접 로그인/스크래핑 없음 (법적 리스크 회피)
3. 공개/문서화된 API 우선 사용, 스크래핑 없음. 공식 API가 없는 경우 비공식 공개 엔드포인트를 허용하되, 반드시 수동 입력 폴백을 함께 구현. 부동산은 사용자가 호가 기준으로 수동 입력.
4. MVP는 2-3주 내 배포 가능한 범위
5. 부부 공유 기능은 Day 1 필수

## Premises

1. 기존 마이데이터 앱이 부동산(전세/매매), 금, 코인, 우리사주 등 한국 가구의 복잡한 자산 구조를 제대로 커버하지 못한다.
2. 맞벌이 부부 간 자산 현황 공유가 되지 않아 재무 의사결정에 비효율이 생긴다.
3. MVP는 잘 설계된 수동 입력(종목/수량) + 시세 자동 반영으로 검증. PMF 확인 후 증권사 연동에 투자. (세컨드 오피니언 도전 수용, 창업자 동의)

## Cross-Model Perspective

독립적인 Claude subagent의 콜드 리드:

- **Steelman:** 한국 맞벌이 부부를 위한 가구 단위 자산 총사령부. "한국형 자산 분류 체계"(전세/매매 구분, 우리사주 베스팅, IRP/ISA 세제)가 차별화 해자.
- **핵심 인사이트:** "15-20분 걸리지만 하고 있다" → 실시간 대시보드가 아닌 "주간 스냅샷 + 변동 알림"이 진짜 사용 패턴.
- **도전:** 전제 3(자동 연동 필수) → "수동 입력이라도 3분이면 5배 개선. 어차피 전세, 금, 우리사주는 API 없음." → 창업자 수용하여 전제 수정.
- **48시간 프로토타입:** Next.js + Supabase + Vercel. 부부 초대, 카테고리별 수동 입력, 순자산 한 화면, 주간 변동 알림.

## Approaches Considered

### Approach A: 주간 스냅샷 (Minimal Viable)
완전 수동 입력. 부부가 주 1회 "재무 체크인" 루틴으로 3분 안에 전체 자산 업데이트. 주간 변동 알림.
- Effort: S (1주) | Risk: Low
- Pros: 즉시 배포, 10가구에 빠른 테스트
- Cons: 수동 입력 피로 → 리텐션 불확실

### Approach B: 반자동 대시보드 (선택됨)
수동 입력(종목/수량) + 공개 API로 시세 자동 반영. 한번 종목과 수량을 설정하면 금액은 실시간 계산.
- Effort: M (2-3주) | Risk: Med
- Pros: 설정 후 시세 자동, 수동 입력 최소화, 대부분 자산에 공개 API 존재
- Cons: API 통합 복잡도, 일부 자산(전세보증금 등)은 여전히 수동

### Approach C: 부부 재무 플래너 (확장형)
대시보드 + 목표 설정 + 자산 배분 추천 + 세금 최적화. 자산관리를 넘어 재무 계획 도구.
- Effort: L (1-2개월) | Risk: High
- Pros: 차별화 강력, 유료화 근거
- Cons: 범위 크리프, 초기 검증 지연

## Recommended Approach

**Approach B: 반자동 대시보드**

종목/수량만 입력하면 시세가 자동 반영되는 구조. 마이데이터 없이도 "한번 설정, 자동 업데이트"에 가까운 경험 제공.

### 핵심 기능

**1. 가구 대시보드 (메인 화면)**
- 상단: 총 순자산 숫자 (크게) = 총자산 - 총부채
- 주간/월간 변동 표시 ("지난주 대비 +2.3%")
- **자산 유형별 파이 차트** (부동산 / 주식 / 연금 / 금 / 코인 / 현금)
- **투자 자산 배분 파이 차트** (국내주식 / 해외주식 / 채권 / 원자재 / 현금성). 올웨더 포트폴리오 스타일 자산 배분 분석. 주식/ETF 종목을 자산 클래스로 자동 분류.
- 자산 목록 (카테고리별 그룹핑, 각 항목에 현재가 + 변동)
- 부채 섹션 (대출 잔액, 연결된 부동산 표시)

**2. 자산 등록 흐름**
- 카테고리 선택 → 세부 입력
  - 부동산: 유형(소유/전세), 주소 입력 + 호가 기준 금액 수동 입력 (네이버 부동산/호갱노노 참고), 전세보증금 수동
  - 주식: 증권사 + 계좌유형(연금/ISA/일반) + 종목 검색 + 수량 → 시세 자동 → 자산 클래스 자동 분류
  - 금: 보유 그램 → 금 시세 API 자동
  - 코인: 종목 + 수량 → 코인 시세 API 자동
  - 우리사주: 종목 + 수량 → 시세 자동 (일반 주식과 동일)
  - 연금/IRP: 금액 수동 입력 (상품별 시세 추적 어려움)
  - 현금: 은행명 + 금액 수동 입력 (예적금, CMA, 파킹통장 등)
  - 부채: 유형(주택담보/신용/학자금/기타) + 잔액 + 금리(선택) + 연결 자산(선택). 순자산에서 차감.

**3. 부부 공유**
- 가구(household) 개념. 한 사람이 가구 생성 → 파트너 이메일 초대
- 각자 자기 자산 입력, 대시보드에서 합산 + 개인별 구분 가능
- 권한: 가구원이면 가구 내 모든 자산/부채를 조회 + 편집 가능. owner_user_id는 입력자 기록용.

**4. 주간 리포트**
- 매주 월요일 이메일/푸시: "이번 주 순자산 변동 요약"
- 어떤 자산이 올랐고/내렸는지 한눈에

**5. "숫자 업데이트" 루틴**
- 자동 시세가 안 되는 자산(연금, 전세보증금, 현금, 부채)만 업데이트 가이드
- "업데이트할 항목 N개 → 예상 소요시간 N분"

### 기술 스택

- **Frontend:** Next.js + TypeScript + Tailwind CSS + shadcn/ui
- **Backend:** Supabase (Auth, PostgreSQL, Row Level Security)
- **배포:** Vercel
- **시세 API:**
  - 주식: FinanceDataReader 라이브러리 (KRX/네이버금융 기반, Python) 또는 Next.js에서 직접 네이버금융 비공식 API 호출. 한국투자증권 Open API는 계좌 필요하므로 2순위.
  - 코인: Upbit REST API (공개, 문서화 우수, 1순위 연동 대상)
  - 부동산: 사용자 수동 입력 (호가 기준). 네이버 부동산/호갱노노 참고 링크 제공.
  - 금: 한국금거래소 시세 or Naver 금 시세 비공식 API
  - 해외주식: Yahoo Finance API + 환율은 한국은행 환율 API 자동 변환
- **시세 연동 우선순위:**
  - Phase 1 (1주차): 인증 + 가구/자산 CRUD + 수동 입력 전체 + Upbit(코인) + 금 시세 연동
  - Phase 2 (2주차): 국내주식 시세 + 해외주식/환율 + 자산 클래스 자동 분류 + 투자 배분 차트 + 주간 리포트
  - Phase 3 (3주차): 폴리싱 + 부채 입력 + PWA + 10가구 배포
  - 부동산은 수동이므로 API 불필요.
- **API 장애 대응:** API 실패 시 마지막 성공 시세를 표시하고 "N시간 전 기준" 라벨 표시. 허용 가능 지연: 주식/코인 1일, 금 3일.
- **차트:** Recharts 또는 Nivo

### 인증 및 초대 흐름

- **인증:** Supabase Auth, 이메일/비밀번호 + 카카오 소셜 로그인 (한국 사용자 편의)
- **파트너 초대:** 가구 생성자가 파트너 이메일 입력 → Supabase에서 초대 토큰 포함 이메일 발송 → 파트너가 링크 클릭 시 회원가입/로그인 후 가구에 자동 합류
  - 가구 최대 인원: 2명 (부부)
  - 초대 토큰 만료: 7일. 만료 시 재발송 가능.
  - 이미 다른 가구에 속한 사용자: 초대 거부 + "이미 가구가 있습니다" 안내
  - 중복 초대: 기존 토큰 무효화 후 새 토큰 발송
- **세션:** Supabase의 JWT 기반 세션. Refresh token으로 자동 갱신.

liability_snapshots

household_snapshots
  - id, household_id, total_assets, total_liabilities, net_worth, snapshot_date
  - (asset_snapshots 생성 직후 가구 단위 합산. 트렌드 차트 쿼리용.)
  - MVP 규모(10가구)에서는 per-asset 스냅샷으로도 충분하지만,
    가구 단위 롤업을 Day 1부터 만들어두면 6개월 트렌드 쿼리가 단순해짐.

### 자산 클래스 자동 분류

종목 입력 시 자동으로 투자 자산 배분 카테고리를 부여. 사용자가 수정 가능.

**분류 로직 (우선순위순):**
1. **ETF 이름 키워드 매칭:** "미국채", "국채", "채권" → bond / "S&P500", "나스닥", "미국" → foreign_equity / "금", "골드", "원유", "원자재" → commodity / "머니마켓", "단기채" → cash_equiv
2. **종목 섹터 정보:** KRX 종목은 KOSPI/KOSDAQ 섹터 코드 → domestic_equity. 해외 종목은 거래소 기준 → foreign_equity.
3. **수동 오버라이드:** 사용자가 자산 상세에서 클래스를 직접 변경 가능.

**비주식 자산 매핑:**
- 부동산 → alternative
- 금 현물 → commodity
- 코인 → alternative
- 현금/예적금 → cash_equiv
- 연금/IRP → 내부 구성 모르면 기본 domestic_equity, 사용자 수정 가능

### 시세 갱신 주기

"주간 체크인" 패턴이 실제 사용 패턴 (세컨드 오피니언 인사이트). 실시간 스트리밍이 아님.
- **자동 시세 갱신:** 1일 1회 배치 (Supabase Edge Function, 매일 18:00 KST)
- **수동 편집 반영:** Supabase Realtime으로 파트너에게 즉시 반영 (부부 동시 접속 시)
- **온디맨드 갱신:** 사용자가 "새로고침" 누르면 최신 시세 즉시 조회

### 데이터 생명주기

- **멤버 탈퇴:** 해당 멤버의 자산은 가구에서 분리. 본인만 접근 가능.
- **가구 해산:** 양쪽 동의 시 가구 삭제. 각자 자산 데이터는 개인 계정에 보존.
- **계정 삭제:** 모든 개인 데이터 즉시 삭제 (한국 개인정보보호법/PIPA 준수). 가구 내 공동 자산(shared)은 파트너에게 소유권 이전 후 삭제.

## Open Questions

1. 연금/IRP 잔고 자동 업데이트 방법? (현재로선 수동이 유일)
2. 세금 전/후 자산 표시? (연금 계좌는 세전 금액이지만 인출 시 과세)
3. 부채(대출) 추적 범위? (MVP에서는 부동산 담보대출만 지원하고, 신용대출 등은 후순위로 제안)

## Success Criteria

1. 창업자 가구가 4주 연속 주 1회 이상 접속하여 자산 현황 확인
2. 10가구 배포 후 4주 리텐션 30% 이상

## The Assignment

**지인 맞벌이 부부 3쌍에게 카카오톡으로 메시지를 보내세요.**

"지금 자산 현황 어떻게 관리하고 있어? 나는 증권사 앱을 6개나 열어봐야 해서 귀찮은데, 혹시 너희도 그래?"

이 질문에 대한 답이 제품의 방향을 결정합니다. "나도 그래" + 구체적인 불편함이 나오면 첫 번째 테스트 사용자 확보. "별로 안 불편한데"가 나오면 타겟 재검토 신호.

## What I noticed about how you think

- 자산 목록을 줄 때 증권사, 계좌유형, 소유자까지 구분해서 적었습니다. "키움 - 연금 - 아라", "키움 - 연금2개 - 희성". 이건 이미 데이터 모델링을 하고 있는 겁니다. 사용자가 아니라 설계자의 시선.
- "수동은 안 된다"고 즉시 전제 3에 이의를 제기했습니다. 사용자 경험에 대한 직감이 있다는 뜻입니다. 편의성이 리텐션의 핵심이라는 걸 알고 있습니다.
- 세컨드 오피니언의 "MVP는 수동으로" 제안을 수용했습니다. 확고한 의견이 있으면서도 더 나은 논리 앞에서 유연하게 전환. 이건 드문 조합입니다.
