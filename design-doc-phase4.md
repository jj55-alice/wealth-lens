## Problem Statement
사용자(부부 가구)가 토스/네이버 증권 같은 일반 증권 앱에서는 받기 어려운 **"내 보유 비중 + 매수가 컨텍스트가 반영된 인사이트"** 를 원함. 단순 뉴스 피드가 아니라, "내가 가진 엔비디아가 매수가 대비 -15%인데 오늘 새 부정 뉴스가 나왔다 → 위험" 같은 개인화된 신호.

## What Makes This Cool
- 일반 증권 앱은 모든 사용자에게 같은 뉴스를 똑같이 보여줌. wealth-lens는 **이 사용자의 포트폴리오**를 알기 때문에 "당신에게만 중요한 뉴스 5건"을 LLM이 골라서 한 줄로 요약 가능.
- 매수가, 보유 비중, 자산 클래스 분포까지 컨텍스트로 함께 넘겨서 "리밸런싱 관점에서 추가 매수 후보" 같은 액션 제안 가능.
- 매일 아침 한 화면에 정리되는 "오늘의 포트폴리오 브리핑" — 출근 전 1분이면 충분.

## Constraints
- 솔로 사이드 프로젝트, 가구 1~2명만 사용 → 인프라 비용 최소화 (Anthropic API 비용 포함 월 $5 미만 목표).
- 이미 있는 인프라 활용: KIS sync로 보유 종목 실데이터, KRX/Yahoo 시세, household 단위 owner 분리, Vercel + Supabase.
- 한국 시장 + 미국 시장 둘 다 (현재 사용자 자산: 국내 ETF + 미국 빅테크).
- DESIGN.md = Industrial/Utilitarian, 카드 위주, 색상은 의미있을 때만, tabular-nums 강제.

## Premises
1. **사용자가 매일 아침 한 번만 봐도 충분하다.** 실시간 알림은 과함. 일일 cron으로 충분. (agree)
2. **개인화된 한 줄 요약 + 액션 태그가 원본 뉴스 전체를 읽는 것보다 가치 있다.** 사용자는 시간 부족, 신호만 원함. (agree)
3. **종목 상세 모달은 on-demand 호출로 충분하다.** 미리 생성하지 않고 사용자가 클릭할 때 LLM 호출 + 1일 cache. (agree)
4. **Anthropic Claude (Haiku 또는 Sonnet)** 가 비용/품질 균형 최적. Sonnet으로 시작해서 비용 보고 Haiku로 다운 가능. (agree)
5. **금융 조언이 아니라 "정보 정리"** 라는 disclaimer 필요. "이건 투자 권유가 아닙니다." (agree)

## Recommended Approach: B-lite (브리핑 카드 + 측정 인프라 — MVP)

**CEO 리뷰 후 조정:** 모달은 MVP에서 제외 → 1주 dogfood 후 데이터 보고 결정. 그 자리에 측정 인프라(피드백 버튼, 비용 로깅, 실패 알림, 뉴스 폴백)를 채움.

## Recommended Approach: B (브리핑 카드 + 종목 상세 모달, 원본 — 차후 단계)

### 데이터 흐름
```
[매일 06:00 cron]
  → 보유 종목 ticker 목록 조회
  → 종목별 최근 24시간 뉴스 fetch (네이버 + Yahoo)
  → 포트폴리오 컨텍스트 (매수가, 비중, 수익률)와 함께 Claude 호출
  → 인사이트 카드 N개 생성 (위험/기회/중립 태그 + 한 줄 요약 + 액션)
  → briefing_cards 테이블에 저장 (날짜별 1 row per household)
  → 대시보드 상단에 표시

[종목 상세 클릭]
  → 모달 오픈
  → 최근 7일 뉴스 5건 표시 (cache 1일)
  → "이 종목 분석" 버튼 → on-demand LLM 호출 → 강세/약세/관망 + 근거
  → 결과는 ticker별 cache (briefing_analyses 테이블)
```

### 데이터 소스
- **국내 뉴스**: 네이버 금융 종목 뉴스 (`https://m.stock.naver.com/api/news/stock/{ticker}` — RSS 또는 JSON 검증 필요. Phase 1에 직접 fetch 테스트.)
- **해외 뉴스**: Yahoo Finance v8 API의 quoteSummary news 모듈 또는 공개 RSS (`https://feeds.finance.yahoo.com/rss/2.0/headline?s={ticker}&region=US&lang=en-US`).
- **공시 (옵션, Phase 2)**: DART API. 사용자가 OPEN DART API key 등록 시.

### LLM 호출 디자인
- **모델**: `claude-haiku-4-5-20251001` (저렴, 빠름) 또는 `claude-sonnet-4-6` (고품질). 시작은 Sonnet, 비용 보고 Haiku 다운 가능.
- **호출 횟수**: 일일 브리핑 = 가구당 1회/일. 종목 상세 = ticker당 최대 1회/일 (cache).
- **System prompt**: 한국어 출력. JSON output (cards 배열). disclaimer 강제.
- **비용 추정**: Sonnet 기준 1회 호출 ~$0.01. 가구 2명 × 30일 × 종목 14개 모달 호출 = 월 ~$1~2.
- **SDK**: Anthropic 공식 `@anthropic-ai/sdk` 또는 Vercel AI SDK. 현재 프로젝트가 Vercel 위에 있으니 **Vercel AI SDK v6** 권장 (스트리밍, 타입세이프, 모델 swap 쉬움).

### DB 스키마 (B-lite, MVP)
```sql
-- 마이그레이션 20260408000004_briefing.sql

create table briefing_cards (
  id uuid primary key default gen_random_uuid(),
  household_id uuid not null references households(id) on delete cascade,
  generated_at timestamptz not null default now(),
  date date not null default current_date,
  cards jsonb not null,  -- [{ticker, name, signal: 'risk'|'opportunity'|'neutral', headline, context, news_urls, feedback?: 1|-1|null}]
  model text not null,
  status text not null default 'success' check (status in ('success', 'partial', 'failed', 'empty')),
  -- 측정 인프라 (CEO 리뷰 추가)
  input_tokens int,
  output_tokens int,
  cost_usd numeric(10,6),
  error_message text,
  unique (household_id, date)
);

create index idx_briefing_cards_household_date on briefing_cards(household_id, date desc);

-- RLS: household members only
alter table briefing_cards enable row level security;

create policy "Household members can view briefing cards"
  on briefing_cards for select
  using (household_id in (
    select household_id from household_members where user_id = auth.uid()
  ));

create policy "Household members can update briefing feedback"
  on briefing_cards for update
  using (household_id in (
    select household_id from household_members where user_id = auth.uid()
  ));

-- briefing_analyses (모달 차후 단계 — Phase B에서 추가)
```

## Approaches Considered (Phase 4)

### Approach A (이미 office-hours에서 논의됨)
일일 브리핑 카드만. 모달 없음. 가장 작음. CC 2-3시간.
- Pros: 가장 빠른 ship, 즉각 가치
- Cons: 사용자가 "더 깊이 보고 싶다"는 욕구 충족 안 됨

### Approach B (선택됨) ★
브리핑 카드 + 종목 상세 모달. CC 반나절.
- Pros: 능동적 탐색 가능, A 인프라 그대로 재사용, 단계적 ship 가능 (먼저 카드만, 그 다음 모달)
- Cons: 모달 컴포넌트 추가 작업, on-demand cache 로직 복잡도 약간

### Approach C
대화형 챗봇. CC 1일+.
- Pros: 가장 자유도 높음
- Cons: 환각 위험, 책임 이슈, 사용 빈도 불확실, AI 비용 가장 큼. 너무 이름.

## Open Questions
1. **뉴스 fetch 안정성**: 네이버 종목 뉴스 endpoint가 안정적인지 직접 검증 필요 (이전에 dividends.ts 네이버 endpoint가 죽어있던 전례 있음). RSS 폴백 + 캐시 강제 필요할 수도.
2. **다국어**: 미국주식 뉴스는 영어 → LLM이 한국어로 요약. 토큰 + 비용 ↑ 약간. 허용 범위.
3. **알림 (이메일/푸시)**: MVP 제외. 사용자가 매일 직접 대시보드 들어가서 확인. 차후 Resend로 추가 가능.
4. **리스크 면책 표시 위치**: 카드 하단 footnote? 또는 첫 사용 시 한번 모달? — footnote가 덜 거슬림.
