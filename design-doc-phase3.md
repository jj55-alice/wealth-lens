## Problem Statement

Wealth Lens는 한국 가구 자산 관리 대시보드로, 부동산/주식/코인/금/현금을 통합 추적한다. 현재 "보기만 하는 앱"이라 자산을 보여주지만 액션을 취할 수 없다. 사용자는 자산을 "키우기 위한" 리밸런싱 제안과, 장기적으로 키움 Open API를 통한 직접 매매를 원한다.

동시에 코드베이스에 타입 안전성 문제 6건, 에러 처리 누락 8건+, UX 갭 7건, 성능 이슈 4건이 발견되어 기존 품질 개선이 선행되어야 한다.

## What Makes This Cool

한국에서 "가구 단위 자산 추적 + 리밸런싱 제안"을 하는 앱이 없다. 뱅크샐러드/토스는 개인 단위, 리밸런싱 없음. M1 Finance/Wealthfront는 미국 전용. Wealth Lens는 이 빈 공간에 있다.

특히 가구원별 리스크 프로파일(공격형/안정형)을 고려한 "가구 단위 최적화"는 개인 앱에서는 절대 못하는 영역.

## Constraints

- 한국 시장 타겟 (국내 증권사, KRX, 네이버 금융 등)
- Next.js 16 + Supabase 스택
- 2인 가구 중심
- 개인 개발자 유지보수
- 기존 KB 부동산 API 폐쇄 상태 (직방으로 교체 완료, 시세 정확도 확인 필요)

## Premises

1. 현재 앱의 가장 큰 병목은 "보기만 하는 앱"이라는 점이다 — 리밸런싱 제안이 "보는 앱 → 하는 앱" 전환의 핵심
2. 리밸런싱 MVP는 목표 비율 설정 → 현재 vs 목표 비교 → 조정 제안이면 충분하다 — 자동 매매는 나중
3. 키움 Open API는 리밸런싱과 별개로 독립 추가 가능
4. 기존 기능 완성도(에러 처리, 타입 안전성, UX)를 리밸런싱과 병행해야 한다 — 기반이 흔들리면 새 기능도 신뢰 못함
5. 개인/가족 용이지만 서비스화 가능성을 열어두고 만든다

## Cross-Model Perspective

Claude 서브에이전트의 독립 분석:

**가장 멋진 버전: "Family War Room"** — 가구원별 리스크 프로파일 설정 + 가구 전체 포트폴리오가 목표에 도달하도록 "누가 무엇을 사고팔아야 하는지" 계산. 세금 시뮬레이션(양도세, 금투세, "지금 하면 세금 얼마 vs 1월에 하면 얼마")까지 붙이면 한국에 이런 도구는 없다.

**핵심 인사이트:** 사용자가 말한 "키우기 위한"이 핵심. 리밸런싱 기능이 아니라 "내 대시보드가 나한테 돈을 벌어다 주는 느낌"을 원하는 것.

**참고 오픈소스:** Ghostfolio — 포트폴리오 분석, 자산배분 시각화, TWR/MWR 계산 패턴 참고 가능.

## Approaches Considered

### Approach A: 리밸런싱 먼저
- Summary: 리밸런싱 MVP를 바로 만들고 기존 버그는 나중에
- Effort: S (CC ~30분)
- Risk: Med — 기반이 흔들리는 상태에서 새 기능 추가
- Pros: 새 기능이 빨리 보임, 동기부여
- Cons: 타입 에러/에러 처리 누락이 리밸런싱 계산에도 영향, stale 데이터로 잘못된 리밸런싱 제안 가능

### Approach B: 품질 다져놓고 리밸런싱 (선택됨)
- Summary: 타입 안전성 + 에러 처리 + stale 경고 등 기반 개선 → 리밸런싱 MVP
- Effort: M (CC ~1.5시간)
- Risk: Low — 탄탄한 기반 위에 새 기능
- Pros: 신뢰할 수 있는 데이터 위에 리밸런싱, 기존 사용 경험도 개선
- Cons: 새 기능까지 시간이 좀 더 걸림
- Reuses: 기존 allocation-pie-chart, asset-list 컴포넌트, Recharts

### Approach C: 가구 리밸런싱 풀버전
- Summary: B + 가구원별 리스크 프로필 + 세금 시뮬레이션
- Effort: L (CC ~2시간)
- Risk: Med — 세금 계산 정확도 검증 필요
- Pros: 한국에 없는 유일한 도구, 서비스화 시 최대 차별점
- Cons: 범위가 큼, 세금 로직 복잡도

## Recommended Approach

**Approach B** — 품질 다져놓고 리밸런싱 MVP.

기반 개선은 리밸런싱의 정확도를 직접 좌우한다. stale 데이터 경고 없이 리밸런싱 제안을 하면 위험하고, 에러가 조용히 삼켜지면 잘못된 포트폴리오 비율이 계산된다.

## 실행 계획

### Phase A: 기반 품질 개선 (CC ~1시간, best-case ±50%)

#### A-1. 타입 안전성 (Critical)
- `AssetWithPrice` 인터페이스의 타입 미스매치 감사 — 각 `as unknown as` 캐스팅이 무엇을 우회하는지 파일별로 확인 후 근본 원인 수정
- 대상 파일 8곳: `stock-portfolio.tsx`, `asset-list.tsx`, `stock-treemap.tsx`, `dashboard/page.tsx`, `upbit-sync/route.ts`, `settings/route.ts`, `queries.ts`, 기타 확인 필요
- 참고: `purchase_price`는 `Asset` 타입에 이미 존재할 수 있음 — 실제 타입 정의 확인 후 누락된 필드만 추가

#### A-2. 에러 처리 개선 (Critical)
- 불충분한 에러 처리(console.log만 있거나 사용자 피드백 없는 catch 블록) → console.error + 사용자 토스트 알림으로 개선
- 환율 조회 실패 시 "환율 정보 없음" 경고 표시 (`dashboard/page.tsx:156`)
- API 동기화 실패 시 구체적 에러 메시지 (`upbit-sync`, `kis-sync`)

#### A-3. Stale 데이터 경고 (High)
- 7일 이상 된 시세에 경고 배너 추가 (`asset-list.tsx`)
- 가격 출처 배지: 🔄 API, 📌 수동, ⏰ 추정가 (`stock-portfolio.tsx`, `asset-list.tsx`)

#### A-4. 수익률 계산 통합 (Medium)
- `src/lib/calculations.ts` 유틸리티 생성
- `stock-portfolio.tsx`, `asset-list.tsx`, `returns/page.tsx` 3곳의 중복 로직 통합

#### A-5. 성능 개선 (Medium)
- 스냅샷 쿼리에 날짜 범위 필터 추가 (`monthly-change.tsx`)
- 해외주식 없으면 환율 조회 스킵 (`dashboard/page.tsx`)

#### A-6. UX 소소한 개선
- 목표 프로젝션: 음수 성장률 처리 (`goal-projection.tsx`)
- 빈 allocation chart 처리
- 로딩 스피너 추가 (동기화 버튼)

### Phase B: 리밸런싱 MVP (CC ~30분, best-case ±50%)

#### B-1. DB 스키마
```sql
CREATE TABLE rebalancing_targets (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  household_id UUID REFERENCES households(id) ON DELETE CASCADE,
  asset_class TEXT NOT NULL, -- domestic_equity, foreign_equity, bond, commodity, crypto, cash_equiv (real_estate는 Open Question #1 해결 후 추가)
  target_ratio DECIMAL(5,2) NOT NULL, -- 0.00 ~ 100.00
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(household_id, asset_class)
);
```

#### B-2. 목표 비율 설정 UI
- `/settings` 페이지에 "리밸런싱 목표" 섹션 추가
- 자산 클래스별 슬라이더 바 (색상: 기존 COLORS 배열 매칭) + 우측 퍼센트 표시
- 합계 100% 실시간 검증 (초록=100%, 빨강=불일치, 저장 버튼 비활성화)
- UI + API 둘 다 100% 검증
- 슬라이더 증감 단위: 1%
- 프리셋 버튼 (4개):
  - 보수형: 국내30 해외20 채권20 원자재10 크립토5 현금15
  - 균형형: 국내25 해외20 채권15 원자재10 크립토10 현금20
  - 공격형: 국내30 해외25 채권5 원자재5 크립토20 현금15
  - 직접설정: 모든 값 수동
- 모바일: 슬라이더 터치 + 숫자 직접 입력 모드 병행

#### B-3. 현재 vs 목표 비교 화면 (/rebalancing 페이지)
- **정보 계층:**
  1. 히어로: 유동 자산 총액 (큰 글씨) + 상태 배지 (균형/조정필요/긴급)
  2. 도넛 차트 2개: 현재 배분 vs 목표 배분 (Recharts 재사용, 기존 allocation-pie-chart 패턴)
  3. 제안 카드 리스트 (가장 큰 갭부터)
  4. 액션 버튼: "목표 수정" + "다시 계산"
- **상태 배지 기준:** 모든 클래스 ±3%p 이내=초록(균형), 하나라도 7%p+=빨강(긴급), 나머지=노랑(조정 필요)
- 모바일: 도넛 차트 grid-cols-2 → sm 이하에서 세로 스택 (grid-cols-1)
- "마지막 확인: N일 전" nudge (localStorage)
- **진입점:** 대시보드 allocation-pie-chart 아래에 요약 카드 ("리밸런싱: 조정 필요 3건" + 상세보기 링크)

#### B-3.5. 인터랙션 상태 (전체)
| 상태 | 사용자가 보는 것 |
|---|---|
| 로딩 | Skeleton 패턴 (기존 대시보드와 동일) |
| 목표 미설정 (첫 방문) | "리밸런싱 목표를 설정해보세요" + "목표 설정하기" CTA (따뜻한 empty state) |
| 자산 없음 | "등록된 자산이 없어요" + "자산 등록하기" CTA |
| 균형 상태 (제안 0건) | "포트폴리오가 균형 잡혀 있어요!" + 초록 체크 아이콘 (긍정 피드백) |
| Stale 30%+ | 노랑 경고 배너: "자산의 N%가 7일 이상 지난 시세입니다. 제안은 참고용입니다." + 금액 옆 "추정" 라벨 |
| 에러 | 토스트 알림 + 재시도 안내 |

#### B-4. 리밸런싱 제안 엔진
- `src/lib/rebalancing.ts` 순수 함수 + `/api/rebalancing/suggest` API 래퍼 (Phase C 매매 대비)
- 자산 클래스 단위 조정 제안: "국내주식에서 340만원 매도, 현금성 자산에 520만원 이동"
- **제안 카드 디자인:** 좌측(클래스명 + 현재% → 목표%), 우측(금액 + 방향). 매도=red-500, 매수=emerald-500
- **Stale 데이터 정책:** 자산 클래스 내 30% 이상이 stale(7일+)이면 경고 표시 + 마지막 알려진 가격으로 계산 + disclaimer 포함
- **환율:** USD 자산은 /api/exchange-rate 환율로 KRW 환산 후 계산
- 종목 단위 제안은 다음 단계 (Phase C에서)
- 카드 UI로 제안 표시 (가장 큰 갭부터 정렬)

#### B-4.5. 자산 → 클래스 매핑
현재 자산 데이터를 리밸런싱 클래스에 매핑하는 규칙:
| asset.type | 조건 | asset_class |
|---|---|---|
| stock | market = KRX (코스피/코스닥) | domestic_equity |
| stock | market = NYSE/NASDAQ | foreign_equity |
| crypto | - | crypto |
| gold | - | commodity |
| cash, savings, deposit | - | cash_equiv |
| real_estate | - | real_estate (TBD) |
| bond | - | bond |
| other | - | 사용자 지정 또는 other |

#### B-5. 리밸런싱 히스토리 (Phase B에서는 간소화)
- "마지막 리밸런싱 확인: 45일 전" nudge만 구현 (DB 없이 localStorage)
- 승인/거부 로그 테이블은 Phase C로 연기

## What I noticed about how you think

- "자산을 키우기 위한" — 단순 추적이 아니라 능동적 자산 운용을 원한다. 대시보드가 돈을 벌어다 주는 도구가 되길 바란다.
- "둘 다 병행" — 기존 완성도와 새 기능 사이에서 타협하지 않겠다는 의지. 둘 다 하겠다는 건 야심이 아니라 기준이 높다는 뜻.
- "전부 다" — 10x 비전에서 하나도 빼지 않았다. 장기 로드맵에 대한 확신이 있다.
- Approach B(품질 먼저)를 선택한 것 — 새 기능의 유혹보다 기반의 중요성을 아는 판단력.
